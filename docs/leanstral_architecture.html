Technical Architecture Specification

LeanEcon: From Proof of Concept to Prototype

A formal verification pipeline for mathematical claims in economics, built on Lean 4, Mathlib, and Leanstral.

v0.4 · March 19, 2026 · Internal working document

01 The Core Idea

An economist writes a paper claiming: "Under CRRA utility, relative risk aversion is constant and equal to γ." This claim has a mathematical proof — but the proof lives in the author's head, the referee's trust, and perhaps a few lines of algebra in an appendix. Nobody machine-checks it.

LeanEcon changes this. The system takes the claim, translates it into a formal theorem in Lean 4 (a dependently typed proof assistant), generates a machine-checkable proof, and verifies it against Lean's kernel. If the kernel accepts the proof, the claim is mathematically certified from axioms. Not LLM confidence. Not statistical likelihood. Certainty.

The Curry-Howard Correspondence
In Lean 4, proofs are programs and propositions are types. A theorem statement is a type; a proof is a value inhabiting that type. When lake build succeeds with no errors and no sorry, Lean's type checker has verified that the proof term correctly inhabits the proposition type — every logical step checked from the axioms of dependent type theory. This is the same foundational principle that makes compilers reject ill-typed programs. The verification is deterministic and independent of the model that generated the proof.

The system has three trust layers, each with a fundamentally different reliability model:

Figure 1 — Trust Model

This separation is the architectural invariant. Leanstral generates candidate proofs (stochastic, may fail). The economist validates the claim is correctly stated (human judgment). Lean's kernel verifies the proof (deterministic, infallible within its axiom system). No component trusts another.

02 Architectural Evolution

Phase 1: Proof of Concept (Sessions 1–4)

The POC asked a minimal question: can we get Lean 4 to verify a claim from an economics paper, end-to-end? The answer was yes — for claims we already knew how to formalize.

Figure 2 — POC Architecture (template-based)

Bottleneck: the template stage. Each claim type (CRRA, elasticity, bifurcation) had a hand-written Python function that produced the Lean theorem. The tool could only verify claims it already knew how to formalize — making it a demo, not a tool.

Verified: CRRA constant RRA, Stone-Geary constant ΔV. Skipped: bifurcation (Mathlib coverage gap).

Phase 2: Prototype — Current Architecture

Templates replaced with LLM-based formalization. Leanstral handles both translation (claim → Lean theorem) and proving (theorem → tactic proof). A pre-classification step screens unformalizable claims before wasting API calls.

Figure 3 — Prototype Architecture (Leanstral-based)

04 Target Architecture: lean-lsp-mcp Integration

The current prototype uses Leanstral as a one-shot proof generator via the Mistral chat completions API. This works, but it's not how Leanstral was designed to operate. Per Mistral's documentation, Leanstral was "specifically trained to achieve maximal performance with the frequently used lean-lsp-mcp" — a Model Context Protocol server that gives the model live access to Lean's compiler during proof generation.

The Performance Gap
Leanstral's benchmark scores (FLTEval) were measured WITH lean-lsp-mcp providing real-time feedback. Our one-shot architecture is leaving performance on the table. The feedback loop in the current prototype is a budget approximation of what lean-lsp-mcp provides natively.
Figure 5 — Target Architecture (lean-lsp-mcp integration)
Status Update
The MCP-first agentic prover described here is now operational in the repo. This section remains as the architecture rationale for the implementation now shipping in LeanEcon.

The target architecture replaces the batch prove-then-verify pattern with an agentic loop where Leanstral interacts with Lean's compiler via MCP tools on every tactic step. This is how Leanstral achieves its benchmark results — and it's how human Lean users work too, constantly checking the goal state after each tactic in the Lean InfoView panel in VSCode.

Available MCP tools that would integrate into the loop:

ToolPurposeWhen used
lean_goalGet proof goal at a file locationAfter each tactic, to see what remains
lean_diagnostic_messagesGet errors, warnings, infosAfter each tactic, to catch failures immediately
lean_multi_attemptScreen multiple tactic snippetsWhen choosing between approaches (rpow_add vs simp)
lean_leansearchNatural language search for Mathlib lemmasWhen stuck — "how to simplify rpow expressions"
lean_proofs_completeCheck if all proofs in a file are doneAfter each tactic, to know when to stop
lean_completionsCode autocompletion for tactics/termsWhen constructing tactic sequences
Implementation Path
lean-lsp-mcp is a Python package (uvx lean-lsp-mcp) that runs as an MCP server. Leanstral already supports MCP via function calling in the Mistral API. The integration requires: (1) running lean-lsp-mcp pointed at our lean_workspace, (2) registering the MCP tools in our Leanstral API calls, (3) replacing the one-shot prove→verify loop with an agentic loop that calls tools between tactic steps. This is a substantial architectural change — estimated at 2–3 build sessions — but the components exist and are documented.

05 Verification Results and Known Boundaries

ClaimDomainFormalizationVerificationNotes
CRRA constant RRA Micro ✓ 1 attempt ✓ 1 attempt Field arithmetic. Clean.
Stone-Geary constant ΔV Macro/Culture — (raw Lean) ✓ 1 attempt Log terms cancel via ring_nf.
Budget constraint Micro ✓ 1 attempt ✓ 1 attempt Biconditional with existential. Elegant.
Cobb-Douglas elasticity Macro ✓ 1 attempt ⚠ 0/5 rpow residual goal. Feedback loop expected to fix.
Second welfare theorem Micro/GE Rejected — Correctly classified as REQUIRES_DEFINITIONS.

What works well

Claims that reduce to field arithmetic (involving +, -, *, /, ⁻¹) or ring arithmetic with Real.log formalize and verify reliably, often on the first attempt. The sorry-validation loop catches bad formalizations before they waste proving attempts. The pre-classifier prevents unformalizable claims from entering the pipeline at all.

What doesn't work yet

Claims involving Real.rpow with variable exponents (K^α, c^(-γ)) formalize correctly but fail at the proving stage because Leanstral can't see the post-tactic goal state. The error feedback loop (currently being implemented) is the direct fix. Claims requiring definitions not in Mathlib (welfare theorems, Nash equilibrium, Arrow's theorem) are correctly rejected but cannot be verified — this is a fundamental Mathlib coverage constraint, not a LeanEcon limitation.

Boundaries of the approach

Lean 4 can verify any constructive mathematical statement, but Mathlib's current coverage determines what's practically verifiable. The strongest results will come from algebraic economics: utility theory, production functions, demand systems, budget constraints, elasticities, comparative statics. Claims requiring measure theory, stochastic calculus, fixed-point theorems, or game-theoretic solution concepts are beyond current scope — not because Lean can't express them, but because the supporting library infrastructure doesn't exist yet.
